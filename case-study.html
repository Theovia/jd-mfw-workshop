Migración ERP completa + capa de inteligencia operativa con IA

Empresa de fabricación y kitting con inventario en dos ubicaciones. El proyecto tuvo dos fases: migrar toda la operación a un ERP nuevo, y construir una aplicación de inteligencia operativa sobre él.

Fase 1 / Migración ERP

Formalizar el modelo de negocio y migrar 5,688 productos

El sistema anterior — una base de datos legacy más hojas de cálculo manuales — no soportaba la complejidad real de la operación: productos que llegan en bulto y se transforman en presentaciones de venta (armado, fraccionamiento, reempaque), trazabilidad de lotes con fechas de caducidad, y costos calculados a partir de ratios de transformación.

El proyecto no fue solo migrar datos. Fue entender y formalizar el modelo de negocio completo, construir la infraestructura de validación, y entregar un sistema operativo funcional desde día uno.

5,688 productos migrados
3,004 listas de materiales
49 decisiones formalizadas
40+ scripts de transformación

Contrato de verdad de dominio. Antes de escribir código, documenté un contrato formal de 7 partes que define todas las reglas de negocio: cómo se clasifican los productos, qué transformaciones son válidas, qué está prohibido inventar (códigos, precios, códigos de barras), y qué decisiones requieren aprobación del dueño. Las 49 decisiones del proyecto están numeradas, documentadas y trazables a una cláusula del contrato.

Modelo híbrido de manufactura. La operación tiene 4 rutas de transformación (compra directa, armado/kit, fraccionamiento y reempaque), cada una con reglas distintas de BOM, costeo y unidades de medida. Construí la lógica de clasificación automática basada en análisis de ratios para que el ERP generara las órdenes de manufactura correctas sin intervención manual.

Pipeline de validación de 5 niveles. Cada producto pasa por verificación de códigos únicos, ratios positivos, precios no negativos, referencias cruzadas entre ubicaciones y normalización de formato. El 32% de los productos tenía costos faltantes; construí una estrategia de recuperación escalonada (4 niveles) que resolvió 1,800+ casos automáticamente. Solo 6 productos requirieron captura manual.

Integración operativa completa. Además de la migración, construí un sistema de pre-tickets con 35 funciones edge que conectan una app móvil con el ERP en tiempo real. La operación procesa ~50 transferencias internas al día; sin este sistema el ERP iría 2-3 horas detrás de la realidad. Once flujos de trabajo rediseñados (compras, recepción, manufactura, transferencias, ventas minorista y mayoreo, mermas, etc.).

Dos implementaciones paralelas. Mantuve una versión de producción en vivo (pragmática, funcionando desde la semana 2) y una versión formal con base de datos intermedia en SQLite (auditable, reproducible). Si algo fallaba en producción, podía reconstruir el sistema completo en menos de 4 horas desde los archivos fuente.

Resultado: 10 de 10 flujos verificados end-to-end. Go-live en ~4 semanas. 268,928 transacciones históricas analizadas para validar la estrategia de códigos de barras. 100% de cobertura de barcode en 8,727 productos finales.

Fase 2 / Inteligencia operativa

Una aplicación completa sobre el ERP como capa de inteligencia

Migrar datos a un ERP nuevo resuelve el problema de infraestructura. Pero la operación del día a día — decisiones de compra, priorización de inventario, ofertas de proveedores, transferencias entre ubicaciones — necesita una capa de inteligencia que el ERP no tiene.

Construí una aplicación web completa que funciona como sistema nervioso de la operación: conecta el ERP con inteligencia artificial, notificaciones en tiempo real y lógica de negocio específica para cada rol del equipo.

12+ pantallas especializadas
21 funciones serverless
7 roles de usuario
13 tablas con seguridad por fila

Búsqueda inteligente. El equipo de mostrador necesita encontrar productos rápido. Construí un motor de búsqueda con coincidencia difusa, autocorrección y más de 50 sinónimos del dominio (un cliente pide “paracetamol” y el sistema encuentra la presentación correcta aunque el nombre comercial sea distinto). Cero dependencia de que el usuario escriba el nombre exacto.

Priorización de inventario por caducidad. Un panel dedicado (VendePrimero) clasifica todo el inventario por fecha de caducidad con semáforo visual: rojo (<30 días), naranja (<60), amarillo (<90). El sistema sugiere descuentos escalonados para mover producto antes de que expire. Lo que antes era una revisión manual de miles de productos ahora es una lista priorizada con acción sugerida.

Motor de alternativas genéricas. Cuando un producto no está disponible, la app sugiere automáticamente alternativas con el mismo principio activo, ordenadas por fecha de caducidad (FEFO). El equipo de mostrador no necesita memorizar el catálogo completo.

Bot de Telegram con IA para ofertas de proveedores. Los proveedores envían ofertas por WhatsApp/Telegram como imágenes, PDFs, mensajes de texto. Construí un bot que recibe esos mensajes y usa Gemini para parsear ofertas no estructuradas en JSON normalizado: producto, precio, descuento, vigencia. El equipo de compras recibe las ofertas ya procesadas, comparables y listas para decidir.

Scoring de proveedores y pronóstico de demanda. Cada proveedor acumula un score de confiabilidad (0-100) basado en entregas, precios y cumplimiento. La app usa Gemini 2.5 Flash con 3 años de historial de ventas para proyectar demanda y generar sugerencias de compra. No reemplaza el criterio del comprador; le da datos que antes no tenía.

Sistema de pre-tickets en tiempo real. Cualquier persona en mostrador puede generar un pre-ticket desde la app que se sincroniza instantáneamente con el punto de venta del ERP (vía funciones edge que crean cotizaciones en el sistema). El cajero carga la cotización, cobra, y el inventario se actualiza. Sin este puente, había una desconexíon de 2-3 horas entre lo que se vendía y lo que el sistema reportaba.

Arquitectura. React + TypeScript en el frontend, Supabase (PostgreSQL + Edge Functions en Deno) como backend, sincronización bidireccional con el ERP vía API REST, notificaciones en tiempo real por WebSocket, caché local con TTL de 10 minutos para funcionar sin conexión estable. Cada rol del equipo (mostrador, cajero, bodega, compras, dueño, operaciones, contabilidad) ve solo la información relevante para su función.

Esta aplicación no es un dashboard de reportes. Es infraestructura operativa que el equipo usa todos los días para tomar decisiones.
